讲解Go中的goroutine

一、什么是goroutine

其实就是go中的协程，但是这个协程在操作系统中并不是真实存在的，那么它是怎么运行的呢，听我娓娓道来

有了线程，那为什么又要多一个协程呢？这都是我们的疑惑

goroutine的本质

在go的源码中可以搜索g

可以看到协程的本质就是一个结构体

与线程的对比

线程是操作系统所能操作的最小单元，而协程只是一个结构体，相比之下协程就会比线程小很多的资源，而且线程的创建，删除，调度都会大量的消耗系统资源，而协程是多个在线程中循环进行，其切换所消耗的资源几乎可以忽略不记，这就是go为什么天然支持高并发的原因

本质也了解了，对比也对比完了，那么我们就来看看g这个结构体中有哪些重要字段

goroutine重要字段

  • stack：它存储这个协程的栈信息

    •   type stack struct {
  	lo uintptr
  	hi uintptr
  }

  • atomicstatus：协程的状态

  • m：当前运行在哪个线程上

又出现一个问题，在go中对线程是怎么描述的，通过搜索底层代码，发现他也是一个结构体m

线程的抽象

其中的g0，是我们的g0协程，去操作调度器的

还有一个curg，当前正在运行的协程g

二、协程如何在线程上运行的

单线程循环

我们的协程是在线程上循环的去运行的，当然这是最早的模型，其中的goexit()是被强制插入进去的

当然上面的是单线程循环，还有我们的多线程循环

多线程循环

一个更抽象的图

我们线程每次会去全局的队列抓取协程，当多线程之后就会有并发的问题，这是就需要加锁，这是go早版本的的调度模型，加锁之后性能就会有问题，所以go对这个模型进行了一个改进

还有一个问题，在线程上的协程只能同步的去运行，无法并发

三、协程调度模型改进（GMP调度模型）

这时就引出了著名的GMP调度模型，G就是协程，M就是线程，P是新引入的东西，为了调度，也是一个结构体

此时引入了本地队列的思想，既然我执行完就要抓取，那我一次抓取直接抓多个

在一个线程里面有一个队列，在外部有一个全局的队列

在p中有一个本地队列

引入了p之后，大大减小了并发冲突的可能性

当我们的p中的协程执行完毕之后，就需要去全局队列中拿协程，当我们的全局队列中没有协程的时候怎么办呢，就让这一个线程在这空着吗，不，p帮我们做了一个优化，他会去看其他的线程的本地队列中有没有协程，有的话直接从其中去拿一些，提高了go的并发性

还有一个优化点，新建协程，随机寻找一个p，直接放到p的runnext上，插一下队，如果p的本地队列满了，那么就放到全局的队列上去

但是现在还有一个问题，只有当上一个协程运行完的时候才能切换下一个协程，此时就需要用到操作系统中的线程执行概念了，使用时间片轮转或者主动挂起
